# GitHub Issue テンプレート

## 機能タスク

```markdown
## 概要

[機能の説明。何を実現するか、なぜ必要かを2〜3文で]

## 受け入れ条件

- [ ] [具体的な条件1]
- [ ] [具体的な条件2]
- [ ] テストが書かれていてパスする
- [ ] `pnpm check` が通る

## 実装メモ

- このタスクは `/feature-dev` スキルで実装できます
- テストは Vitest（ユニット）/ Playwright（E2E）を使ってください
- 関連する設計方針は `docs/architecture.md` を参照してください

## 参考

- [関連するdocsへのリンク]
- [関連Issue]
```

---

## 環境セットアップタスク

```markdown
## 概要

[何をセットアップするか]

## 作業内容

- [ ] [手順1]
- [ ] [手順2]

## 完了条件

- [ ] `pnpm check` が通る
- [ ] CI が green になる

## 実装メモ

- このタスクは `/node-project-setup` スキルで実行できます（Node.jsプロジェクトのセットアップの場合）
```

---

## 調査タスク

```markdown
## 概要

[何を調査・決定する必要があるか]

## 背景

[なぜこれを決める必要があるか。どのタイミングまでに決定が必要か]

## 調査観点

- [ ] [観点1: 例) パフォーマンス面での比較]
- [ ] [観点2: 例) コスト・運用負荷の比較]
- [ ] [観点3: 例) チームの習熟度]

## 完了条件

- [ ] 決定事項を `docs/architecture.md` または `docs/requirements.md` に記録する
- [ ] 関連するIssueを更新する（必要であれば新規Issueを作成する）
```

---

## CI/CDタスク

```markdown
## 概要

GitHub Actions を使って CI/CD パイプラインを構築する。

## 作業内容

- [ ] PR時のチェックワークフロー作成（lint / type-check / test）
- [ ] main マージ時のデプロイワークフロー作成
- [ ] 必要なシークレットの設定（リポジトリ設定で実施）

## 完了条件

- [ ] PRを作成した際にCIが自動で走る
- [ ] mainへのマージ時にデプロイが自動で走る
- [ ] ワークフローのバッジをREADMEに追加する
```

## TBD（技術選定・方針の先送り）タスク

```markdown
## 概要

[何を決定する必要があるか。例: DBをMySQL / PostgreSQLどちらにするか]

## 背景

[なぜ今決められなかったか。例: チームの意見が割れている、PoC結果待ち]

## 決定期限

YYYY-MM-DD までに決定する（理由: [マイルストーン名]の前に必要なため）

## 検討観点

- [ ] [観点1: 例) パフォーマンス面での比較]
- [ ] [観点2: 例) コスト・運用負荷]
- [ ] [観点3: 例) チームの習熟度]

## 完了条件

- [ ] 決定事項を `docs/architecture.md` に記録する
- [ ] `CLAUDE.md` の技術スタックセクションを更新する
- [ ] この Issue を受けて発生する実装 Issue を作成する
```
