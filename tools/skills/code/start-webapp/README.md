# start-webapp

新規Webアプリプロジェクトの立ち上げを対話的にサポートするウィザード。
ヒアリング→設計ドキュメント作成→タスク分解→GitHub Projectへのkanban登録まで一貫して行う。

## 使い方

```
/start-webapp
```

## 前提条件

- 対象リポジトリがすでに作成済みで、カレントディレクトリがそのリポジトリルートであること
- `gh` CLI がインストール・認証済みであること
- 環境構築（Node.js・DB等のローカル環境）はユーザーが実施する

## 動作

### Phase 1: ヒアリング（6ステップ）

各ステップで確認を取りながら進める:

1. **プロジェクト概要・ゴール** — 解決したい課題、リリース期限
2. **ターゲットユーザー・主要機能** — Must/Shouldの機能洗い出し
3. **技術スタック選定** — フロントエンド・バックエンド・DB・認証・インフラなどの候補提示と選択
4. **設計方針** — データモデル・認証・API設計・テスト方針
5. **ファーストリリーススコープ確定** — MoSCoW法で機能を整理
6. **未決事項の洗い出し** — 調査タスクとしてIssue候補に追加

### Phase 2: 設計ドキュメント作成

- `CLAUDE.md` — プロジェクト概要・技術スタック・開発規約
- `docs/architecture.md` — 設計方針・主要コンポーネント・データモデル
- `docs/requirements.md` — 要件一覧・MoSCoWスコープ

### Phase 3: GitHub Project登録

- `gh project create` でカンバンボードを作成
- 環境セットアップ・CI/CD・認証・各機能・調査タスクをIssueとして作成しProjectに登録
- 1タスク = 1PRで完結できる粒度
- タスクの説明文にはSkillの活用ヒントを記載（例: `/feature-dev` で実装できる）

### Phase 4: 進捗に応じたタスク追加

Should機能の実装開始前・調査結果が出た時など、都度ユーザーと確認しながらIssueを追加する。

## 例

```
/start-webapp
```

> こんにちは。まず、このアプリで解決したい課題やゴールを教えてください。
> ファーストリリースの期限や優先度があれば合わせて教えてください。

（ユーザーが回答）

> ありがとうございます。次に、主なユーザー像と、ファーストリリースで必ず提供したい機能を教えてください。
> 「ログイン機能が欲しい」といった場合は、誰が・どの権限で・どのデバイスで使うかも合わせて教えてください。

...（以降 Step 3〜6 → Phase 2 ドキュメント作成 → Phase 3 GitHub Project登録へ続く）

## タスク粒度の方針

| タスク種別       | 例                                                               |
| ---------------- | ---------------------------------------------------------------- |
| 環境セットアップ | `chore: TypeScript・リンター設定` → `/node-project-setup` で実行 |
| 機能実装         | `feat: ユーザー認証の実装` → `/feature-dev` で実装               |
| 調査             | `調査: DBスキーマ設計方針を決定する`                             |
| インフラ         | `chore: GitHub Actions CI/CDの構築`                              |
