# GitHub Issue テンプレート（既存プロジェクト向け）

## ドキュメント補完タスク

```markdown
## 概要

[何のドキュメントを補完するか。例: `POST /users` エンドポイントのリクエスト/レスポンス仕様が未記載]

## 背景

[なぜドキュメントが必要か。例: 新規エンジニアがAPIの仕様を理解できず、コードを読み解くのに時間がかかっている]

## 作業内容

- [ ] [対象のコード・APIを読んで仕様を把握する]
- [ ] [ドキュメント（README / architecture.md / コードコメント）に追記する]
- [ ] 追記内容が実装と一致していることを確認する

## 完了条件

- [ ] 対象のAPI・機能の仕様が文書化された
- [ ] 既存の記述と矛盾がない
- [ ] 新規エンジニアがコードを読まずに理解できる内容になっている

## 実装メモ

- このタスクは `/implement-task` スキルで実装できます
- 変更後は `/doc-sync` スキルで実装と文書の同期を維持することを検討してください
- 関連する設計方針は `docs/architecture.md` を参照してください
```

## テスト追加タスク

```markdown
## 概要

[どのモジュール・機能にテストを追加するか。例: `src/payments/` モジュールにユニットテストが存在しない]

## 背景

[なぜテストが必要か。例: リファクタリング時にリグレッションが発生しやすく、変更が怖い状態になっている]

## 作業内容

- [ ] [対象モジュールのテスト対象を洗い出す]
- [ ] ユニットテストを追加する（正常系・異常系・境界値）
- [ ] 必要に応じて統合テスト・E2Eテストを追加する

## 完了条件

- [ ] 主要な処理パスにテストが存在する
- [ ] テストがすべてパスする（`pnpm test` または該当のテストコマンドが green）
- [ ] CIでテストが自動実行される

## 実装メモ

- このタスクは `/implement-task` スキルで実装できます
- 既存のテストファイルのパターンに合わせてください（テストフレームワーク・ディレクトリ構成）
```

## 技術的負債解消タスク

```markdown
## 概要

[何を解消するか。例: `UserService` が1000行を超えており、複数の責務が混在している]

## 背景

[なぜ負債が問題か。例: 機能追加のたびに影響範囲が読めず、変更コストが高い]

## 作業内容

- [ ] [現状の問題を把握する（影響範囲・依存関係を確認）]
- [ ] [分割・整理の方針を決める]
- [ ] [実装する]
- [ ] 既存のテストがすべてパスすることを確認する

## 完了条件

- [ ] 既存の挙動が変わらない（テストで確認済み）
- [ ] [具体的な完了基準。例: ファイルが400行以下に収まった / 重複コードが解消された]
- [ ] `pnpm check`（またはlint・型チェックコマンド）が通る

## 実装メモ

- このタスクは `/implement-task` スキルで実装できます
- リファクタリング前にテストが不十分な場合は、先にテスト追加タスクを完了させることを推奨します
- 関連する設計方針は `docs/architecture.md` を参照してください
```

## セキュリティ修正タスク

```markdown
## 概要

[何のセキュリティ問題を修正するか。例: `config.ts` に API キーがハードコードされている]

## 背景

[なぜ問題か。例: コードリポジトリに秘密情報が含まれており、漏洩リスクがある]

## 作業内容

- [ ] [問題箇所を特定する]
- [ ] [修正を実施する。例: 環境変数に移行し、`.env.example` に追記する]
- [ ] 修正後に既存の動作が壊れていないことを確認する

## 完了条件

- [ ] 秘密情報がソースコードにハードコードされていない
- [ ] `.env.example` が更新されている（環境変数の場合）
- [ ] `pnpm check` が通る

## 実装メモ

- このタスクは `/implement-task` スキルで実装できます
- 修正後は `git log` や `git diff` でコミット履歴に秘密情報が残っていないか確認してください
```

## CI/CDタスク

```markdown
## 概要

[何のCI/CD問題を修正・追加するか。例: GitHub Actions の型チェックステップが失敗し続けている]

## 作業内容

- [ ] [問題の原因を特定する]
- [ ] [修正または追加を実施する]
- [ ] PRを作成して CI が green になることを確認する

## 完了条件

- [ ] 対象のCIジョブが green になる
- [ ] PRごとに自動でチェックが走ることを確認した

## 実装メモ

- このタスクは `/implement-task` スキルで実装できます
- ワークフローファイルは `.github/workflows/` を参照してください
```

## 調査タスク

```markdown
## 概要

[何を調査・決定する必要があるか。例: `legacy-auth.ts` が現在も使われているか確認し、廃止計画を立てる]

## 背景

[なぜ今決める必要があるか。例: 触れないままでいるとオンボーディングの障壁になり続ける]

## 調査観点

- [ ] [観点1: 例) 現役の使用箇所をコードベース全体で確認する]
- [ ] [観点2: 例) 代替実装が存在するか確認する]
- [ ] [観点3: 例) 廃止する場合の影響範囲を洗い出す]

## 完了条件

- [ ] 決定事項を `docs/architecture.md` または `docs/requirements.md` に記録した
- [ ] 次に必要なアクション（廃止・維持・置き換え）のIssueを作成した（必要な場合）

## 実装メモ

- `/subagent-strategy` が利用可能な場合は、大規模なコードベース調査の委譲戦略を参照するとよい
```
