# 依存関係の解析ガイド

## 解析の流れ

1. 全 Issue の本文・コメントを取得する
2. 明示的な依存シグナルを抽出する
3. 技術的な暗黙の依存を推論する
4. リスト外の依存 Issue を追加取得する
5. 循環依存を検出する
6. トポロジカルソートで実行順序を決定する

---

## 明示的な依存シグナルの抽出

### 英語パターン

以下の正規表現で Issue 本文・コメントを検索する:

```
(depends on|blocked by|requires|after|prerequisite:?)\s*#(\d+)
```

### 日本語パターン

```
#(\d+)\s*(に依存|の後に|が前提|が必要|を先に|をベースに)
(前提|依存|ブロック).*#(\d+)
```

### GitHub 自動リンク

Issue 本文内の `#N` 参照のうち、文脈から依存関係を示すものを抽出する。
単なる言及（「#N で議論した通り」など）は依存関係に含めない。

---

## 技術的な暗黙の依存の推論

明示的なシグナルがない場合でも、以下のパターンから依存関係を推論する:

| パターン                             | 例                                           | 判断                           |
| ------------------------------------ | -------------------------------------------- | ------------------------------ |
| DB スキーマ → アプリケーションコード | #10「テーブル追加」→ #12「API 追加」         | #12 は #10 に依存              |
| ライブラリ導入 → 利用コード          | #5「axios 導入」→ #7「API クライアント実装」 | #7 は #5 に依存                |
| 型定義 → 利用コード                  | #3「型定義追加」→ #8「コンポーネント実装」   | #8 は #3 に依存                |
| インフラ設定 → アプリケーション      | #1「CI 構築」→ #2「テスト追加」              | 依存しない（独立して実装可能） |

**判断基準**: Issue A の変更が存在しないと Issue B のコードがコンパイル・実行できない場合のみ依存とする。
「あると便利」レベルの関連は依存に含めない。

---

## リスト外 Issue の追加

指定されたリスト内の Issue が、リスト外の Issue に依存している場合:

1. 依存先の Issue を `gh issue view` で取得する
2. Issue がオープンかつ未対応であることを確認する
3. リストに追加し、ログに記録する

```
追加: #10 — #12 の本文に "depends on #10" と記載されていたため
```

依存先の Issue が既にクローズ済み（実装済み）の場合は追加不要。
該当する変更が `main` に存在するか `gh pr list --search "closes #N" --state merged` で確認する。

## 循環依存の検出

### 検出方法

依存グラフの深さ優先探索（DFS）で循環を検出する。

訪問中のノードに再度到達した場合、循環が存在する:

```
検出例: #10 → #12 → #15 → #10（循環）
```

### 循環検出時の対応

1. 循環に含まれるすべての Issue をリストアップする
2. ユーザーに報告し、どの依存を解消すべきか確認を求める
3. 循環が解消されるまで、該当 Issue の実行を保留する

循環に含まれない Issue は先に実行を進めてよい。

## トポロジカルソート

### アルゴリズム

Kahn のアルゴリズム（入次数ベース）で実行順序を決定する:

1. 各ノード（Issue）の入次数（依存される数）を計算する
2. 入次数 0 のノードをキューに入れる
3. キューからノードを取り出し、実行順序に追加する
4. そのノードが依存先となっている Issue の入次数を 1 減らす
5. 入次数が 0 になったノードをキューに追加する
6. キューが空になるまで繰り返す

### 同一レベルの並び順

入次数が同じ（依存関係上同レベルの）Issue は、Issue 番号の昇順で実行する。

### 出力形式

```
実行順序:
1. #10 (入次数 0, 依存なし) → base: main
2. #11 (入次数 0, 依存なし) → base: main
3. #12 (入次数 1, depends on #10) → base: feat/10-db-schema
4. #15 (入次数 1, depends on #12) → base: feat/12-api-endpoint
```
