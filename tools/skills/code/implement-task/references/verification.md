# 検証ガイド

## 検証の優先順位

以下の順で検証コマンドを探し、見つかった時点でそれを使用する。

### 優先度 1: CLAUDE.md

`CLAUDE.md` に「よく使うコマンド」や「開発ワークフロー」セクションがある場合、そこに記載されたコマンドを最優先する。

例:

```markdown
## よく使うコマンド

- `pnpm check` — 型チェック・lint・フォーマットを一括実行
- `pnpm test` — テストスイートを実行
```

### 優先度 2: package.json スクリプト

Node.js プロジェクトの場合、`package.json` の `scripts` フィールドを確認する:

```bash
# package.json を読んで scripts を確認
Read /path/to/package.json
```

探すスクリプト名（優先順）:

| 目的         | 探すキー                                |
| ------------ | --------------------------------------- |
| 一括チェック | `check`, `validate`, `verify`, `ci`     |
| テスト       | `test`, `test:unit`, `test:integration` |
| 型チェック   | `check:types`, `type-check`, `tsc`      |
| lint         | `check:lint`, `lint`, `eslint`          |
| フォーマット | `check:fmt`, `format:check`, `prettier` |
| ビルド       | `build`                                 |

### 優先度 3: 言語別デフォルト

`CLAUDE.md` も `package.json` も存在しない、または該当スクリプトがない場合、言語のデフォルトツールを試す。

#### Python

```bash
# テスト
pytest

# 型チェック（存在する場合）
mypy .

# lint
ruff check . または flake8 .
```

#### Go

```bash
# テスト
go test ./...

# ビルド確認
go build ./...

# lint（インストール済みの場合）
golangci-lint run
```

#### Rust

```bash
# テスト
cargo test

# ビルド確認
cargo build

# lint
cargo clippy
```

## 検証失敗時の対処

### エラーの分類

検証コマンドが失敗した場合、まずエラーを分類する:

**自分で修正できるエラー:**

- 型エラー（自分が変更したコードに起因）
- lint エラー（フォーマット・スタイル違反）
- テスト失敗（自分の実装に起因するもの）

**ユーザーに確認が必要なエラー:**

- 環境依存のエラー（依存関係が未インストール、環境変数が未設定など）
- 自分の変更以前から存在していたテスト失敗（`git stash` で確認可能）
- スキーマ変更・マイグレーションが必要なエラー

### 既存の失敗を区別する

自分の変更前から失敗していたテストと、自分の変更で壊れたテストを区別する:

```bash
# 自分の変更をスタッシュして検証
git stash
pnpm test  # 失敗するか確認
git stash pop
```

元々失敗していたテストは修正スコープ外として扱い、その旨をユーザーに報告する。

### 修正後の再検証

エラーを修正した後は、**必ずすべての検証コマンドを再実行する**。1件修正して別の箇所が壊れることがあるため、部分的な再実行では不十分。

## 検証完了の基準

Phase 6（コミット）へ進む前に、以下を全て満たすこと:

- [ ] テストが全通過（または元々失敗していたテストのみ残っている）
- [ ] 型エラーがゼロ
- [ ] lint エラーがゼロ
- [ ] ビルドが成功（ビルドスクリプトがある場合）
