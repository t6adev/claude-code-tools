# コードベース探索ガイド

## 探索の目的

Issue を実装するには、変更対象のコードを正確に特定し、プロジェクト固有の規約・パターンを把握する必要がある。
この文書は Phase 2 での探索戦略を詳述する。

---

## 優先して読むべきファイル

以下のファイルが存在する場合、**必ず最初に読む**。プロジェクト固有の規約を無視した実装はレビューで差し戻されるリスクが高い。

| ファイル               | 目的                                 |
| ---------------------- | ------------------------------------ |
| `CLAUDE.md`            | 開発規約・よく使うコマンド・禁止事項 |
| `docs/architecture.md` | コンポーネント構成・設計方針         |
| `docs/requirements.md` | 機能要件・スコープ                   |
| `docs/CONTRIBUTING.md` | コントリビューション規約             |

---

## 探索ツールの使い分け

### Grep — 文字列・シンボル検索

特定の関数名・クラス名・定数を探す場合に使う:

```bash
# 関数名で検索
Grep pattern="function handleLogin" glob="**/*.ts"

# エラーメッセージで探索（バグ修正時）
Grep pattern="session expired" glob="**/*.{ts,tsx,js}"

# 複数キーワードの組み合わせ
Grep pattern="(auth|login|session)" glob="src/**"
```

### Glob — ファイル構造の把握

ディレクトリ構造やファイル一覧を把握する場合に使う:

```bash
# コンポーネント一覧
Glob pattern="src/components/**/*.tsx"

# テストファイル一覧
Glob pattern="**/*.test.{ts,tsx}"

# 設定ファイル一覧
Glob pattern="**/*.config.{ts,js,json}"
```

### Read — ファイル内容の精読

関連性が確認できたファイルのみ読む。推測で読まない。

## プロジェクト規模別の探索戦略

### 小規模プロジェクト（〜50ファイル）

- CLAUDE.md → Issue キーワードで Grep → 関連ファイルを Read の順で進める
- コンテキスト制限を意識する必要は低い

### 中規模プロジェクト（50〜300ファイル）

1. `CLAUDE.md` / `docs/architecture.md` で全体像を把握
2. Issue キーワードで Grep して候補ファイルを絞る
3. 候補ファイルの上位 5〜10 件を Read
4. 実装に直接関わるファイルのみコンテキストに保持する

### 大規模プロジェクト（300ファイル以上）

コンテキストが肥大化するリスクが高い。以下の方針で対処する:

- **300ファイル超の大規模プロジェクトで、かつ複数の独立した調査が必要な場合のみ**サブエージェントに委譲する。それ以外は Grep/Glob を直接使う
- **読むファイル数に上限を設ける**: 1フェーズで読むファイルは最大 10〜15 件
- **差分最小化原則**: 既存の類似機能のパターンを 1〜2 件読めば十分なことが多い

## バグ修正時の探索

### エラーから逆引きする

エラーメッセージ・スタックトレースが Issue に記載されている場合:

1. エラーメッセージのキーワードで Grep
2. スタックトレースに含まれるファイルパスを直接 Read
3. 呼び出し元を辿って原因箇所を特定する

### 再現条件を特定する

- Issue に再現手順が記載されている場合はその通りに辿る
- テストファイルに類似ケースがないか確認する（既存テストがバグを再現できる場合がある）

---

## 機能追加時の探索

### 類似実装を見つける

「同じような機能がすでにある部分」を探し、そのパターンに倣う:

- 認証機能を追加する → 既存の認証ミドルウェアを探す
- 新しい API エンドポイントを追加する → 既存のルートハンドラを探す
- UI コンポーネントを追加する → 既存の類似コンポーネントを探す

### エントリーポイントを特定する

変更を加える場所（追加・修正するファイル）と、その変更が影響を与える範囲を事前に把握する。

---

## 理解が難しいコードに遭遇した場合

目的・設計意図が読んでも不明瞭な場合、
「何をするか・なぜそう書かれているか・落とし穴は何か」を整理してから実装判断を行う。
（`/explain` が利用可能な場合は、そのスキルの出力形式で整理するとよい）

---

## 探索完了の判断基準

以下がすべて揃った時点で Phase 3（実装計画）へ進む:

- [ ] 変更対象ファイルが特定できた
- [ ] プロジェクト固有の規約（命名規則・テスト方針など）を把握した
- [ ] 類似実装のパターンを把握した（機能追加の場合）
- [ ] 問題の原因箇所を特定した（バグ修正の場合）
