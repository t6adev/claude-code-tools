# レポートテンプレート

`REPORT.md` を親ディレクトリに出力する際、以下の構造に従う。
各セクションの内容は実際の作業内容に合わせて記述する。

## テンプレート構造

以下のセクションを順に記述する。

### 1. タイトル

`# 再現環境レポート: <問題名>`

### 2. 問題の概要

- **現象**: 何が起きているか
- **情報源**: PR URL / コミットハッシュ / ユーザー報告 等
- **疑わしい原因**: パッケージ / 環境 / アプリコード / 不明

### 3. 環境情報

テーブルで以下を記載:

- Node.js バージョン
- pnpm バージョン
- TypeScript バージョン（使用している場合）
- OS

続けて「関連パッケージ（元プロジェクト）」として、パッケージ名とバージョンのテーブルを記載。

### 4. パターン詳細

**パターンごとに** 以下の項目を見出し付きで記載する。これがレポートの最も重要なセクション。

- **目的**: このパターンで何を確認するか
- **構成**: 使用パッケージとバージョン。baseline との差分を明記する
- **再現コードの要点**: `repro.ts` の主要部分を抜粋（コードブロックで記載）
- **実行コマンド**: `pnpm run repro` 等
- **実行結果**: stdout / stderr の抜粋（コードブロックで記載）。エラーメッセージは省略せず全文を記載する
- **判定**: ❌ 再現 / ✅ 正常

### 5. 結果一覧

全パターンをテーブルにまとめる。列は「パターン / 主な変更点 / 判定 / 備考」。

例:

| パターン    | 主な変更点                 | 判定    | 備考                         |
| ----------- | -------------------------- | ------- | ---------------------------- |
| baseline    | （現環境の最小再現）       | ❌ 再現 |                              |
| version-a   | package-a を v2.0.0 に変更 | ✅ 正常 | このバージョンでは発生しない |
| version-b   | package-a を v3.0.0 に変更 | ❌ 再現 |                              |
| no-plugin-x | plugin-x を除外            | ✅ 正常 | plugin-x が原因の可能性      |

### 6. 考察と結論

結果から導かれる原因の仮説を記述する。例:

- package-a の v3.0.0 以降で問題が発生する
- plugin-x との組み合わせが原因
- v2.x 系では正常に動作する

### 7. 推奨アクション

次のステップをチェックリスト形式で記述する。例:

- [ ] package-a のリポジトリに Issue を報告する
- [ ] ワークアラウンド: package-a を v2.x に固定する
- [ ] plugin-x の代替を検討する

### 8. 更新履歴

テーブルで日時と変更内容を記録する。新しいものを上に追記していく。

| 日時             | 変更内容                                           |
| ---------------- | -------------------------------------------------- |
| YYYY-MM-DD HH:MM | 初回レポート作成（baseline, version-a, version-b） |

パターンの追加・結果の変更・結論の修正があるたびに行を追加する。

## テンプレート使用時の注意

- **「パターン詳細」が最も重要なセクション**。第三者がこのレポートだけを見て各パターンの結果を再現・検証できる粒度で記述する
- 実行結果は stdout/stderr の全文ではなく、判断に必要な部分を抜粋する。ただし、エラーメッセージは省略せず全文を記載する
- baseline との差分がないファイル（横展開されたテスト等）は「baseline と同一」と書けばよい
- 更新履歴は上に新しいものを追記していく
