# レポートテンプレート

2種類のレポートを親ディレクトリに出力する。

- **`REPORT.md`**（内部レポート）: ユーザー自身が読むもの。プロジェクト固有の文脈を含む
- **`REPORT-EXTERNAL.md`**（外部レポート）: パッケージ作者やコミュニティが読むもの。問題を一般化し自己完結した形で記述する

---

## A. 内部レポート（`REPORT.md`）

### テンプレート構造

以下のセクションを順に記述する。

#### 1. タイトル

`# 再現環境レポート: <問題名>`

#### 2. 問題の概要

- **現象**: 何が起きているか
- **情報源**: PR URL / コミットハッシュ / ユーザー報告 等（プロジェクト固有の参照をそのまま記載してよい）
- **疑わしい原因**: パッケージ / 環境 / アプリコード / 不明

#### 3. 環境情報

テーブルで Node.js / pnpm / TypeScript / OS のバージョンを記載。
続けて「関連パッケージ（元プロジェクト）」としてパッケージ名とバージョンのテーブルを記載。

#### 4. パターン詳細

**パターンごとに** 以下の項目を見出し付きで記載する。これがレポートの最も重要なセクション。

- **目的**: このパターンで何を確認するか
- **構成**: 使用パッケージとバージョン。baseline との差分を明記する
- **再現コードの要点**: `repro.ts` の主要部分を抜粋（コードブロックで記載）
- **実行コマンド**: `pnpm run repro` 等
- **実行結果**: stdout / stderr の抜粋（コードブロックで記載）。エラーメッセージは省略せず全文を記載する
- **判定**: ❌ 再現 / ✅ 正常

#### 5. 結果一覧

全パターンをテーブルにまとめる。列は「パターン / 主な変更点 / 判定 / 備考」。

#### 6. 考察と結論

結果から導かれる原因の仮説を記述する。

#### 7. 推奨アクション

次のステップをチェックリスト形式で記述する。

#### 8. 更新履歴

テーブルで日時と変更内容を記録する。新しいものを上に追記していく。

### 内部レポートの特徴

- プロジェクト固有の情報（リポジトリ URL、ブランチ名、ファイルパス、PR リンク等）をそのまま含めてよい
- ユーザーが文脈を把握しているため、背景説明は簡潔でよい
- 再現コードは要点の抜粋でよい（全文は各パターンのディレクトリにある）

## B. 外部レポート（`REPORT-EXTERNAL.md`）

### テンプレート構造

#### 1. タイトル

問題を端的に表すタイトルにする。Issue タイトルとしてそのまま使える形が望ましい。

例:

- `` `package-x` v3.0.0 で ESM 環境から `functionY` を呼ぶと TypeError が発生する ``
- `` `vite-plugin-react` v5.0.0 と `@emotion/react` v11.14 の組み合わせで HMR が動作しない ``

#### 2. 概要

問題の一般的な説明を 1-3 文で記述する。特定プロジェクトに依存しない表現にする。

#### 3. 再現環境

テーブルで以下を記載:

- Node.js バージョン
- pnpm バージョン
- OS
- 問題のパッケージとバージョン

#### 4. 再現手順

**このセクションだけで問題を再現できる** 自己完結した手順を記述する。

1. `package.json` の全文を記載する
2. 再現コード（`repro.ts`）の全文を記載する
3. `tsconfig.json` が必要な場合はその全文も記載する
4. 実行コマンドを記載する

読者がコピー&ペーストして即座に再現できる品質にする。

#### 5. 期待される挙動 vs 実際の挙動

明確に対比して記述する:

- **期待される挙動**: こうなるべき
- **実際の挙動**: こうなった（エラーメッセージの全文を含む）

#### 6. 調査結果

どのバージョン・条件で発生するか、しないかをテーブルにまとめる。

例:

| 条件                      | 結果    | 備考                                  |
| ------------------------- | ------- | ------------------------------------- |
| package-x v2.5.0          | ✅ 正常 |                                       |
| package-x v3.0.0          | ❌ 発生 | このバージョンから発生                |
| package-x v3.1.0 (latest) | ❌ 発生 |                                       |
| plugin-y を除外           | ✅ 正常 | plugin-y との組み合わせが原因の可能性 |

#### 7. 既知の Issue・関連情報

関連パッケージの GitHub リポジトリで既存 Issue を検索した結果を記載する。

見つかった場合の例:

| パッケージ | Issue               | ステータス | 概要                                                                     |
| ---------- | ------------------- | ---------- | ------------------------------------------------------------------------ |
| package-x  | owner/package-x#456 | open       | 同一の問題。v3.0.0 で導入されたリグレッションとして報告済み              |
| plugin-y   | owner/plugin-y#789  | closed     | 類似の問題。v2.1.0 で修正済みだが、package-x v3 との組み合わせでは未検証 |

見つからなかった場合は「`<パッケージ名>` のリポジトリで関連する Issue は見つからなかった」と明記する。

#### 8. 考察

- 原因の推定
- 影響範囲の考察（どのようなユースケースで問題になるか）
- 既知の Issue との関連（7. の結果を踏まえて）

#### 9. ワークアラウンド

現時点で取れる回避策があれば記述する。ない場合は「現時点で回避策は見つかっていない」と明記する。

### 外部レポートの特徴

- プロジェクト固有の情報を **含めない**（リポジトリ URL、社内ブランチ名、内部ファイルパス等）
- 問題を「このパッケージの、このバージョンで、この条件のとき、こうなる」という **一般化された形** で記述する
- 再現手順が自己完結している（`package.json` とコードの全文を含む）
- パッケージ作者が Issue として受け取れる品質
- 同じパッケージを使う開発者が知見として参照できる内容

---

## 共通の注意事項

- 内部レポートの「パターン詳細」と外部レポートの「調査結果」は同じデータに基づくが、記述の粒度と対象読者が異なる
- 両レポートは作業の進行に合わせて同時に更新する
- baseline との差分がないファイル（横展開されたテスト等）は「baseline と同一」と書けばよい
- エラーメッセージは両レポートとも省略せず全文を記載する
